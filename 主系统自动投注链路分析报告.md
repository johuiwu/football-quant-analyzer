## 一、完整链路追踪

### 1.1 理论链路（设计意图）

```
前端启动监控 → POST /api/corner/start
  → cornerService.startCornerBackendPolling() → 3秒轮询
    → pollOnce()
      → crawlCornerMatches() → 数据采集
      → computeDelta() → 增量检测
      → evaluateStrategies() → 策略评估
      → [策略触发] → saveCornerTrigger() → 记录历史
      → [自动投注条件满足] → checkDuplicateBet() → generatePendingBet() → betQueue.push()
        → processBetQueue() → cornerBetExecutor.executeBet()
          → 9步投注流程 → 结果验证 → 更新数据库
```

- 角球页面的比赛行结构、赔率元素class、投注弹窗可能都有差异

**严重程度**：🔴 高 — 可能导致找不到匹配的比赛或赔率

### 断点2：赔率匹配策略过于简单

**主系统** ([cornerBetExecutor.js](file:///d:/下载/足球竞彩量化分析系统/足球竞彩量化分析系统/backend/services/cornerBetExecutor.js) L93-L114)：

**问题**：
1. **matchName 格式**：依赖 `match.matchName`（如"TeamA vs TeamB"），但角球页面的比赛名称可能不包含完整队名（可能缩写或使用其他语言）
2. **赔率容差0.05**：角球盘口赔率变化频繁，0.05的容差可能不够
3. **赔率选择器**：`[class*='odd'], [class*='price']` 过于宽泛，可能匹配到非赔率元素
4. **没有方向感知**：策略配置了 betDirection（over/under），但投注时不区分方向，只匹配赔率值

**测试脚本**：先等待有效赔率出现（`waitForFunction`），再点击第一个有效赔率，不依赖精确匹配。

**严重程度**：🔴 高 — 可能点击错误的赔率或找不到赔率

### 断点6：投注结果验证逻辑有误

**主系统** ([cornerBetExecutor.js](file:///d:/下载/足球竞彩量化分析系统/足球竞彩量化分析系统/backend/services/cornerBetExecutor.js) L193-L211)：

```javascript
if (body.includes("Insufficient") || body.includes("不足") ||
    body.includes("余额不足") || body.includes("超过限额")) {
  return "failed";
}
```

**问题**：将"余额不足"归类为 `failed`，导致：
1. `cornerService.js` 将投注记录标记为 `failed`
2. 后续同一比赛+策略的去重检查 `checkDuplicateBet` 会查到 `failed` 记录
3. **同一比赛永远不会再尝试投注**

但余额不足实际上意味着**投注链路畅通**，只是账户没钱。应该区分：
- **链路畅通但余额不足** → 应标记为 `insufficient`（不阻止后续重试）
- **真正的投注失败**（赔率变动、盘口关闭等）→ 标记为 `failed`

**严重程度**：🔴 高 — 余额不足后永远不会再尝试该比赛

## 三、前端配置同步链路分析

### 3.1 配置传递路径

```
前端 SettingsPanel
  → cornerStore.setSettings({ isRealMode, autoBetEnabled, betAmount })
    → POST /api/corner/bet-config { isRealMode, amount, autoBetEnabled }

前端 LiveMonitor 追踪按钮
  → useAppStore.addTrackedMatch(matchId)
    → syncTrackedMatchesToBackend(ids)
      → POST /api/corner/bet-config { trackedMatchIds: ids }
```

### 3.2 问题：配置不同步

**前端 cornerStore.ts** (L305-L311)：
```typescript
if ('isRealMode' in partial || 'betAmount' in partial || 'autoBetEnabled' in partial) {
  const s = get().settings;
  fetch('/api/corner/bet-config', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ isRealMode: s.isRealMode, amount: s.betAmount, autoBetEnabled: s.autoBetEnabled })
  }).catch(() => {});
}
```

**问题**：
1. ❌ **trackedMatchIds 不在此同步** — trackedMatchIds 通过 useAppStore 独立同步，两个 store 的配置可能不一致
2. ❌ **autoBetConfirmRequired 不在前端同步** — 后端有此字段但前端没有对应的配置入口
3. ❌ **没有读取后端配置的初始化** — 前端启动时不会从后端读取已有的 betConfig，可能导致重启后配置丢失

## 四、与测试脚本的对比总结

| 维度 | 测试脚本 (test-auto-bet.cjs) | 主系统 (cornerBetExecutor.js) | 差距 |
|------|-----|-----|------|

## 五、结论与建议

### 结论

**主系统当前无法像测试脚本一样自动执行投注下单**，存在6个关键断点，其中3个为高严重度：

1. 🔴 导航到角球页面而非让球/大小页面 — 投注的盘口类型与策略设计不匹配
2. 🔴 赔率匹配策略不可靠 — 可能点击错误赔率或找不到赔率
3. 🔴 余额不足被标记为失败 — 导致同一比赛永远不再尝试

### 建议

1. **统一导航目标**：根据策略的 betDirection 决定导航到角球页面还是让球/大小页面
2. **增强赔率匹配**：增加方向感知（over/under），使用更精确的选择器
3. **修复结果验证**：将余额不足单独标记为 `insufficient`，不阻止后续重试
4. **增强元素定位**：补充 ID 匹配和弹窗容器内查找
5. **完善弹窗处理**：采用测试脚本的三级弹窗处理策略
6. **同步前端配置**：统一 cornerStore 和 useAppStore 的配置同步逻辑
