## 一、投注执行器（cornerBetExecutor.js）

> 源文件：`backend/services/cornerBetExecutor.js`

### 1.1 完整工作流程（9步）

`executeBet(betData)` 函数实现了在 hga050.com 上执行真实角球投注的完整流程，共9个步骤：

**Step 1 — 检查登录状态**（L27-L29）

```javascript
if (!isLoggedIn()) {
  return { success: false, error: "未登录，请先登录 hga050.com" };
}
```

首先通过 `isLoggedIn()` 标记检查登录状态，此标记由 browserPool 模块维护。

**Step 2 — 获取共享页面并验证**（L33-L59）

```javascript
const page = getSharedPage();
```

获取共享浏览器页面后，执行**双重登录验证**——不仅依赖 `isLoggedIn` 标记，还通过 `page.evaluate` 检测页面DOM特征：
- 检测余额关键词：Balance / 余额 / Credit / 额度
- 检测账户元素：`[class*='user']`, `[class*='account']`, `[class*='balance']`
- 检测登出关键词：Logout / 登出 / 退出

**Step 3 — 导航到角球页面**（L65-L67）

```javascript
await navigateToCornersFast(page);
await sleep(3000);
await handlePopups(page);
```

调用 `navigateToCornersFast` 快速导航到角球市场页面，等待3秒渲染后处理可能出现的弹窗。

**Step 4 — 查找匹配的比赛行**（L70-L85）


匹配策略：**主队名+客队名同时匹配**（精确）→ **完整比赛名匹配**（回退）。

**Step 5 — 查找匹配赔率的投注选项并点击**（L93-L114）

```javascript

for (const el of clickables) {
```

## 二、策略评估器（cornerEvaluator.js）

> 源文件：`backend/services/cornerEvaluator.js`

### 2.1 评估条件体系（5大条件）

`evaluateSingleStrategy(match, strategy)` 函数实现了5层条件判断，全部通过才触发策略：

**条件1 — 启用检查**

```javascript
if (!strategy.enabled) return false;
```

策略必须处于启用状态。

**条件2 — 时间窗口**

```javascript
const currentMinute = match.elapsedMinutes ?? match.currentMinute ?? match.elapsed_minutes ?? 0;
// 中场休息排除
if (currentMinute >= 45 && currentMinute <= 46) return false;
// 超时排除
if (currentMinute > 99) return false;
// 时间窗口
if (currentMinute < strategy.playTimeStart || currentMinute > strategy.playTimeEnd) return false;
```

- 排除中场休息期（45-46分钟）
- 排除超过99分钟的异常数据
- 比赛时间必须在策略配置的 [playTimeStart, playTimeEnd] 范围内

**条件3 — 盘口范围（方向感知）**

```javascript
if (strategy.betDirection === "auto" || strategy.betDirection == null) {
  const absHcp = Math.abs(handicap);
  if (absHcp < strategy.cornerHandicapLower || absHcp > strategy.cornerHandicapUpper) return false;
} else {
  if (handicap < strategy.cornerHandicapLower || handicap > strategy.cornerHandicapUpper) return false;
}
```

- `betDirection=auto`：使用盘口绝对值比较（不区分方向）
- 其他值：使用原始盘口值比较（区分大小盘方向）

**条件4 — 赔率条件**

```javascript
if (odds < strategy.targetOdds) return false;
```

赔率必须大于等于目标赔率阈值。

**条件5 — 比分条件（4种模式）**

```javascript
// 模式A：哨兵值，不做比分限制
if (strategy.leadGoals >= 20) return true;

// 模式B：球差不超过阈值
if (strategy.leadGoals > 0 && (strategy.leadGoalsWeak || 0) === 0
    && goalDiff <= strategy.leadGoals) return true;

// 模式C：弱队领先至少N球，受上限约束
if ((strategy.leadGoalsWeak || 0) > 0
    && goalDiff >= (strategy.leadGoalsWeak || 0)
    && goalDiff <= strategy.leadGoals) return true;

// 模式D：平局检查
if (strategy.leadGoals === 0 && (strategy.leadGoalsWeak || 0) === 0
    && goalDiff === 0) return true;
```

| 模式 | leadGoals | leadGoalsWeak | 含义 |
|------|-----------|---------------|------|
| A | >= 20 | 任意 | 不限制比分 |
| B | > 0 | = 0 | 球差不超过N球 |
| C | > 0 | > 0 | 弱队领先至少N球，且球差不超过上限 |
| D | = 0 | = 0 | 必须平局 |

### 2.2 前后端一致性约束

文件头部明确标注（L2-L4）：

```
【同步提醒】前端 cornerStore.ts:export evaluateMatchForStrategies 须与此逻辑保持一致
修改任一处时请同步更新另一处
```

前端独立实现策略评估以支持实时响应，但逻辑必须与后端保持一致。

### 2.3 批量评估

`evaluateStrategies(match, strategies)` 函数对一场比赛评估所有已启用策略，返回触发的策略ID数组：

```javascript
export function evaluateStrategies(match, strategies) {
  return strategies
    .filter(s => evaluateSingleStrategy(match, s))
    .map(s => s.id);
}
```

```javascript
function simulateBetResult(odds) {
```

## 四、投注服务层（cornerService.js）

> 源文件：`backend/services/cornerService.js`

### 4.1 投注队列机制

```javascript
let betQueue = [];
let isProcessing = false;

async function processBetQueue() {
  if (isProcessing || betQueue.length === 0) return;
  isProcessing = true;
  try {
    const task = betQueue.shift();
    // ... 执行投注
  } finally {
    isProcessing = false;
  }
}
```

- **互斥锁**：`isProcessing` 确保同一时间只处理一个投注
- **FIFO队列**：先入先出，每次处理1个任务
- **自动投注触发条件**（L153-L186）：

```javascript
if (betConfig.isRealMode && betConfig.autoBetEnabled) {
  for (const match of matches) {
    if (betConfig.trackedMatchIds.length === 0 ||
        !betConfig.trackedMatchIds.includes(match.matchId)) continue;
    // ... 去重 → 入队
  }
  processBetQueue();
}
```

三个条件必须同时满足：
1. `isRealMode = true`（真实模式）
2. `autoBetEnabled = true`（自动投注开关）
3. `trackedMatchIds` 非空且比赛在白名单中

### 4.2 去重机制

**Level 1 — 历史记录去重**（checkDuplicateBet）

```javascript
const rows = await query(
  "SELECT id FROM corner_history WHERE match_id = ? AND strategy_id = ? AND bet_status IN ('executed', 'failed')",
  [matchId, String(strategyId)]
);
```

查询 corner_history 表中同一 match+strategy 组合的已执行/失败记录。

**Level 2 — 待投注去重**（generatePendingBet）

```javascript
const existing = await query(
  "SELECT id FROM corner_bets WHERE match_id = ? AND strategy_id = ? AND status = 'pending'",
  [match.matchId || "", String(strategyId)]
);
if (existing && existing.length > 0) {
  return { success: true, skipped: true, reason: "duplicate pending" };
}
```

查询 corner_bets 表中同一 match+strategy 组合的待处理记录。

### 4.3 二次确认机制

```javascript
if (betConfig.autoBetConfirmRequired) {
  await run("UPDATE corner_bets SET status = 'pending_confirm' WHERE id = ?", [task.betId]);
  // 不执行投注，等待用户确认
  isProcessing = false;
  return;
}
```

- 开启后，投注进入 `pending_confirm` 状态
- 用户通过 `confirmBet(betId)` 确认执行，或 `rejectBet(betId)` 拒绝
- 确认后重新加入队列执行

### 4.4 后端轮询与策略触发

## 六、投注安全机制

### 6.1 金额限制

```javascript
const MAX_BET_AMOUNT = parseInt(process.env.MAX_BET_AMOUNT || "1000", 10);
```

- 默认上限1000元，通过环境变量配置
- 手动投注和自动投注均受此限制

### 6.2 去重保护

双重去重机制：
1. **corner_history 表**：同一 match+strategy 组合已有 executed/failed 记录时跳过
2. **corner_bets 表**：同一 match+strategy 组合已有 pending 记录时跳过

### 6.3 二次确认

```javascript
betConfig = {
  autoBetConfirmRequired: false  // 默认关闭
};
```

- 开启后投注进入 `pending_confirm` 状态
- 用户必须通过 `confirmBet` API 手动确认
- 可通过 `rejectBet` API 拒绝

### 6.4 告警机制

```javascript
const ALERT_THRESHOLD = 5;
if (consecutiveFailures >= ALERT_THRESHOLD) {
  console.error("[cornerService] 告警: 连续 " + consecutiveFailures + " 次爬取失败!");
}
```

```javascript
    deltaData.changed = true;
    deltaData.changes = { scoreChanged: true, cornerChanged: true, timeChanged: true };

    const cornerChanged = deltaData.totalCorners !== prev.totalCorners;
    const timeChanged = deltaData.elapsedMinutes !== prev.elapsedMinutes;
    deltaData.changed = scoreChanged || cornerChanged || timeChanged;
    deltaData.changes = { scoreChanged, cornerChanged, timeChanged };
  }

  return deltaData;
}
```

```javascript
const DELTA_FIELDS = ["elapsedMinutes", "homeScore", "awayScore", "totalCorners", "homeCorners", "awayCorners"];

function computeDelta(oldMatches, newMatches) {
  const oldMap = new Map(oldMatches.map(m => [m.matchId, m]));
  const deltas = [];
  for (const nm of newMatches) {
    const om = oldMap.get(nm.matchId);
    if (!om) { deltas.push({ matchId: nm.matchId, field: "added" }); continue; }
    for (const field of DELTA_FIELDS) {
      if (JSON.stringify(om[field]) !== JSON.stringify(nm[field])) {
        deltas.push({ matchId: nm.matchId, field, oldValue: om[field], newValue: nm[field] });
      }
    }
    // 额外检测：cornerOU, cornerHDP, nextCorner
  }
  return deltas;
}
```

```
isRealMode && autoBetEnabled && trackedMatchIds非空?
    │
    ├──→ [否] 仅记录触发，不投注
    │
    └──→ [是] 继续
          │
          ▼ 去重检查
checkDuplicateBet(matchId, strategyId)
    │
    ├──→ [重复] 跳过
    │
    └──→ [不重复] 继续
          │
          ▼ 生成待投注
generatePendingBet() → corner_bets表(status='pending')
          │
          ▼ 入队
betQueue.push({ betId, matchId, strategyId, odds, amount })
          │
          ▼ 二次确认检查
autoBetConfirmRequired?
    │
    ├──→ [是] status='pending_confirm' → 等待用户确认
    │         │
    │         ├──→ POST /api/corner/confirm-bet/:id → 重新入队
    │         └──→ POST /api/corner/reject-bet/:id → status='rejected'
    │
    └──→ [否] 直接执行
          │
          ▼ 执行投注
cornerBetExecutor.executeBet(betData)
    │
    ▼ 9步投注流程
    │
    ├──→ [成功] status='executed'
    └──→ [失败] status='failed'
```

## 十六、关键技术决策

### 16.1 浏览器自动化 vs 纯API

| 维度 | API模式 | DOM模式 |
|------|---------|---------|
| 速度 | 快（HTTP请求） | 慢（页面渲染） |
| 资源 | 低 | 高（浏览器进程） |
| 数据完整性 | 部分字段 | 完整8种盘口 |
| 可靠性 | 依赖uid/ver | 依赖页面结构 |
| 反检测 | 低风险 | 高风险 |

**决策**：API模式优先，DOM模式兜底

### 16.2 共享浏览器 vs 隔离浏览器

| 场景 | 浏览器模式 | 原因 |
|------|-----------|------|
| 监测（轮询） | 共享（browserPool） | 避免重复登录，共享Cookie |
| 赛程获取 | 隔离 | 不影响监控的共享会话 |
| 投注执行 | 共享 | 需要已登录的页面状态 |

### 16.3 轮询策略

| 数据源 | 间隔 | 触发条件 |
|--------|------|----------|
| Gismo match_timelinedelta | 3秒 | 持续轮询，比赛结束自动停止 |
| transform.php（辅数据） | 15秒 | 持续轮询 |
| 后端轮询（cornerService） | 3秒 | 启动监控后持续，无比赛自动暂停 |
| HgCrawler轮询 | 10秒 | 手动启动 |
| 缓存过期 | 30秒 | GET /api/corner/live 时检查 |

**策略引擎特殊触发**：比赛时间进入35-55分钟窗口或角球数变化时，立即调用transform.php获取最新盘口数据。

### 16.4 安全设计

| 机制 | 实现 | 配置方式 |
|------|------|----------|
| 金额限制 | MAX_BET_AMOUNT | 环境变量（默认1000） |
| 去重保护 | corner_history + corner_bets 双表查询 | 内置，不可关闭 |
| 二次确认 | pending_confirm 状态 | autoBetConfirmRequired 开关 |
| 连续失败告警 | 5次阈值 | ALERT_THRESHOLD 常量 |
| 真实模式开关 | isRealMode | POST /api/corner/bet-config |
| 白名单 | trackedMatchIds | POST /api/corner/bet-config |
